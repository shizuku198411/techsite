## はじめに

この記事では、MCP Server シリーズの導入として、そもそも MCP Server とは何か、なぜ使うのか、どんな状況で使うべきかを整理します。

MCP を触り始めると、まず気になりやすいのは次のあたりです。

- MCP Server って何をするものなのか
- API を直接呼ぶのと何が違うのか
- どういうときに導入する価値があるのか
- 毎回自作しないといけないのか

このあたりが曖昧なままだと、実装チュートリアルを読んでも「結局どこで効くのか」が見えづらくなります。

そこで今回は、実装に入る前段として、MCP Server の役割と判断ポイントをざっくり掴めるようにまとめます。

## MCP Serverとは

MCP Server は、LLM クライアントから使えるツールやデータアクセスを、共通のプロトコルで公開するためのサーバです。

たとえば、LLM を使うクライアントが

- 天気予報を取得する
- 社内ドキュメントを検索する
- ファイルを読む
- チケット管理ツールの情報を引く
- 独自業務 API を叩く

といった機能を必要とするとき、それぞれをクライアントごとに個別実装するのではなく、MCP Server として切り出して共有できるようになります。

イメージとしては、

- LLM クライアント
  - ChatGPT / Codex / Claude Desktop など
- MCP Server
  - ツールやリソースを公開する層
- 外部サービス
  - REST API / DB / ファイルシステム / 社内システム

の橋渡しをする役割です。

ASCII でざっくり表すと、次のような関係です。

```text
+-----------------------------+
| LLM Client                  |
| ChatGPT / Codex / Claude    |
+-------------+---------------+
              |
              | MCP
              v
+-------------+---------------+
| MCP Server                  |
| tool definition / auth /    |
| formatting / access control |
+------+------+---------------+
       |      | 
       |      +--------------------+
       |                           |
       v                           v
+------+--------+        +---------+--------+
| External API  |        | Local / Internal |
| Weather API   |        | Files / DB / SaaS|
+---------------+        +------------------+
```

つまり、LLM クライアントが直接いろいろな API やローカル資産に触りにいくのではなく、いったん MCP Server を経由してツールとして扱える形に揃える、というイメージです。

つまり MCP Server は、「LLM が安全に使える道具箱を外に出すための共通インターフェース」と考えるとかなり分かりやすいです。

## なぜ使うのか

MCP Server を使う大きな理由は、LLM クライアントと外部機能の接続方法を共通化できることです。

もし MCP がない場合、ある機能を AI から使いたいときは、クライアントごとに

- API 呼び出しコードを書く
- 認証方法を組み込む
- ツール仕様を定義する
- レスポンス整形を書く

といった実装が必要になりがちです。

これをクライアントごとに繰り返すと、

- 同じロジックの重複
- ツール仕様のズレ
- メンテナンスコストの増加

が起きやすくなります。

MCP Server を使うと、この共通部分をサーバ側へ寄せられます。するとクライアント側は「MCP で接続する」だけで、同じツールを再利用しやすくなります。

特に価値が大きいのは次のような点です。

- LLM クライアントをまたいでツールを共通利用しやすい
- 外部 API の認証や整形ロジックを 1 箇所に寄せられる
- ツールの入出力仕様を安定化しやすい
- 社内向け機能をクライアント非依存で公開しやすい

## どういう状況で使うべきか

MCP Server は、何でもかんでも最初から導入すればよいわけではありません。特に向いているのは、「LLM から使いたい機能を、今後も継続して使い回したい」ときです。

たとえば次のような状況なら、かなり相性がよいです。

### 1. 同じツールを複数の LLM クライアントから使いたい

Codex VS Code Extension でも使いたいし、別の MCP 対応クライアントからも同じ機能を使いたい、というケースです。

この場合、クライアントごとに個別実装するより MCP Server にまとめたほうがきれいです。

### 2. 外部 API を LLM 向けに整形したい

外部 API のレスポンスがそのままでは読みにくいことはかなり多いです。

たとえば、

- 生の JSON が大きい
- 内部コードを日本語へ変換したい
- 余計な項目を落として返したい

といった要求があるなら、MCP Server 側で吸収する価値があります。

### 3. 社内ツールやローカル環境を AI から使いたい

MCP Server は公開 API だけでなく、

- 社内システム
- ローカルファイル
- 独自スクリプト
- 開発支援ツール

のような「そのままでは AI に渡しづらいもの」とも相性がよいです。

特に、既存の REST API がないけど内部処理としては呼び出せる、という場面で便利です。

### 4. クライアントごとの差を減らしたい

「この AI クライアントでは使えるけど、別のクライアントでは使えない」を減らしたいときにも、MCP で公開しておく価値があります。

クライアント固有機能ではなく、MCP という共通レイヤを挟むことで、乗り換えや併用がしやすくなります。

## 逆に、まだ MCP Server を作らなくてよいケース

一方で、次のような場合はまだ自作しなくてもよいことが多いです。

- 一度しか使わない簡単なスクリプト
- そもそも LLM から呼び出す必要がない処理
- 既存の公式 MCP Server や公開サーバで十分足りる場合
- クライアント 1 つだけで、接続先も 1 個だけの小さな検証

たとえば「とりあえずこの API を 1 回試したい」だけなら、まずは普通のスクリプトや curl で十分です。

MCP Server は、再利用や共有の価値が出てくる段階で導入するとちょうどよいです。

## 自作は必要？

ここはかなり気になるポイントですが、結論からいうと「毎回自作する必要はない」です。

まず考える順番としては、次のようにすると分かりやすいです。

1. 既存の MCP Server で足りるか確認する
2. 少し設定変更すれば使えるか確認する
3. 足りない部分だけを自作する

ここでいう「既存の MCP Server」は、すでに誰かが用意している MCP 対応のツール群のことです。

たとえばイメージしやすいものだと、次のような種類があります。

- ファイルを読む、書く、検索するサーバ
- GitHub や issue tracker を操作するサーバ
- ブラウザ操作や Web 検索を行うサーバ
- データベースへ接続して検索するサーバ
- 外部 API をラップして公開しているサーバ

実際に公開されている例としても、Filesystem、GitHub、PostgreSQL、Slack、Google Drive などを扱う MCP Server があります。

つまり、「MCP Server を自作する」というのは毎回ゼロから始めることではなく、まずは使える道具がすでにあるかを確認して、足りないところだけを埋める発想に近いです。

既存のサーバで十分なら、それを使うのが最短です。特に

- ファイル操作
- GitHub 連携
- ブラウザ操作
- 汎用的な API アクセス

のような領域は、すでに使えるものがあることも多いです。

ただし、次のようなケースでは自作する意味が出てきます。

- 日本向けに API を差し替えたい
- 返却形式を自分の用途向けに整えたい
- 社内専用 API や独自業務ロジックをつなぎたい
- 特定の LLM ワークフローに合わせてツール設計したい

今回の weather MCP サーバもまさにこのパターンで、公式チュートリアルの米国向け API をそのまま使うのではなく、日本でも使いやすい Open-Meteo へ差し替えています。

つまり、自作が必要かどうかは「MCP Server という仕組みが必要か」ではなく、「既存のサーバで要件を満たせるか」で判断するのがよいです。

## MCP Server を考えるときの判断基準

迷ったときは、次の 4 つで考えると整理しやすいです。

- その機能は LLM から繰り返し使いたいか
- 複数クライアントで共有したいか
- API やデータを中間で整形したいか
- 既存の MCP Server では足りないか

この 4 つのうち複数に当てはまるなら、MCP Server 化を考える価値がかなりあります。
